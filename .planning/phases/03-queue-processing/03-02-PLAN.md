<objective>
Implement queue monitoring, stale job detection, and cleanup to prevent queue resource exhaustion and provide visibility into queue health.

Purpose: Without monitoring, stuck or stale jobs accumulate in Redis, consuming memory and blocking queue processing. This plan adds: (1) stale job detection via createdAt timestamp, (2) stuck job detection via repeated failures, (3) automatic cleanup or dead lettering of problematic jobs, (4) queue health metrics via IntegrationLog and dashboarding.

Output: `/api/cron/monitor-queues` endpoint for monitoring, database schema updates to track queue job history, and monitoring documentation.
</objective>

  <action>
  Create queue monitoring utility that:
  1. Exports monitorQueue(queueName: string, options?: MonitorOptions) function
  2. Options: { staleThresholdMs?: number, failureThreshold?: number } (defaults: 24h, 5 failures)
  3. For each queue, scans for:
     - Stale jobs (created > 24 hours ago, still waiting/active)
     - Dead jobs (failed > 5 times)
     - Stuck jobs (active > 5 minutes, assume crashed processor)
  4. For each problematic job:
     - Log to IntegrationLog with errorCategory="QUEUE_MONITORING"
     - Include job ID, job type, job state, age in ms
  5. Cleanup strategy:
     - Stale failed jobs: remove from queue after logging
     - Dead jobs (>5 failures): move to dead letter queue (handled by BullMQ's removeOnFail)
     - Stuck active jobs (>5min): log warning but DON'T force remove (job might still be processing)
  6. Returns summary: { staleFailed: number, deadJobs: number, stuckJobs: number }
  7. Use efficient scanning: only check delayed/waiting/active states (not completed/failed - those are auto-cleaned)

  Do NOT: Don't remove jobs without logging first. Don't scan failed jobs excessively (they auto-clean). Don't modify job data.
  </action>
  <verify>File exists with proper structure; functions are importable</verify>
