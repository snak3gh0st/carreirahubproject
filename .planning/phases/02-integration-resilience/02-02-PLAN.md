<objective>
Improve error handling and logging with structured context to enable rapid debugging and operator visibility into integration failures.

Purpose: When integrations fail (even with circuit breakers), operators and developers need clear, structured error context to diagnose issues quickly. Current IntegrationLog table exists but lacks rich context. This plan adds detailed error categorization, structured error metadata, and graceful fallback responses that guide users to recovery actions.

Output: Enhanced IntegrationLog schema, structured error logging for all integrations, graceful fallback messages for user-facing errors.
</objective>

Enhance IntegrationLog table to store rich error context:

1. Add fields to IntegrationLog model in Prisma schema:
   - errorCode (String?, e.g., "EXTERNAL_TIMEOUT", "AUTH_FAILED", "RATE_LIMITED")
   - errorCategory (String?, e.g., "transient", "permanent", "auth", "validation", "unknown")
   - errorSeverity (String?, e.g., "info", "warning", "error", "critical")
   - recoveryAction (String?, e.g., "retry", "manual_intervention", "contact_support", "check_circuit")
   - metadata (Json?, arbitrary structured data: {statusCode, retryAfter, circuitState, etc})
   - durationMs (Int?, API call duration for performance tracking)

2. Update integrationLogger in lib/utils/logger.ts:
   - Add logError(service, action, error, {errorCode, category, severity, recovery, metadata}) method
   - Error type detection: catch errors and extract code (from error.code, error.status, error.message pattern matching)
   - Category mapping:
     * Transient: timeout, ECONNREFUSED, 429 (rate limit), 503 (service unavailable)
     * Permanent: 401 (auth), 403 (forbidden), 404 (not found), 400 (bad request)
     * Auth: 401, token expiration, invalid credentials
     * Validation: 400 bad request, missing fields
     * Unknown: everything else
   - Severity mapping:
     * Critical: auth errors, permanent failures
     * Error: 5xx, timeouts
     * Warning: 4xx, partial failures
     * Info: successful operations with degradation

3. Create createRecoveryAction(errorCategory, service): string method
   - Transient → "retry" (circuit breaker will handle)
   - Permanent → "manual_intervention" (operator needs to investigate)
   - Auth → "check_circuit" (re-authenticate service)
   - Validation → "contact_support" (client error, need to check request)
   - Unknown → "contact_support"

DO NOT change existing IntegrationLog fields (backward compatible).

Update all catch blocks in 8 services to use new structured error logging:

Pattern for each service:
```typescript
try {
  return await cb.execute(async () => {
    // API call
  });
} catch (error) {
  const errorCode = extractErrorCode(error); // e.g., "EXTERNAL_TIMEOUT"
  const category = categorizeError(error);    // e.g., "transient"
  const recovery = createRecoveryAction(category, serviceName);

  await integrationLogger.logError("serviceName", "actionName", error, {
    errorCode,
    category,
    severity: calculateSeverity(category),
    recovery,
    metadata: {
      statusCode: error.status || error.statusCode,
      message: error.message,
      retryable: category === "transient"
    }
  });

  // Return appropriate fallback or throw
  throw error;
}
```

Implement extractErrorCode() helper in each service to handle provider-specific error codes:
- Pipedrive: Extract from response.error.message or use HTTP status code

- DocuSign: error.code or error.message pattern match
- Twilio: error.code or error.status
- RetellAI: error.code from response
- OpenAI: error.status (e.g., 429, 500)
- Resend: error.code from response

DO NOT change method signatures or return types (preserve backward compatibility).
DO NOT log sensitive data (passwords, API keys, tokens) - filter from metadata.
WHY: Backward compatible. Structured logging enables debugging. Consistent patterns across services.

<task type="auto">
  <name>Task 3: Create graceful fallback responses with user-actionable guidance</name>
  <files>lib/utils/error-fallback.ts, app/api/leads/route.ts, app/api/deals/route.ts, app/api/customers/route.ts</files>
  <action>
Create fallback response generator and integrate into user-facing API endpoints:

1. Create lib/utils/error-fallback.ts with function:
   createUserFallbackResponse(serviceName: string, action: string, errorCategory: string): {message: string, recoveryAction: string, estimatedRecoveryTime?: string}

   Response examples:
   ```typescript
   // Transient error (will retry automatically)
   {
     message: "Pipedrive service is temporarily unavailable. Your action is queued and will be retried automatically.",
     recoveryAction: "wait",
     estimatedRecoveryTime: "5 minutes"
   }

   // Permanent error (operator action needed)
   {
     message: "Unable to create invoice (QuickBooks auth expired). Support team has been notified.",
     recoveryAction: "contact_support",
     supportLink: "https://support.example.com/quickbooks-auth"
   }

   // Auth error (user action needed)
   {
     message: "Stripe integration requires re-authentication.",
     recoveryAction: "reconnect_integration",
     actionUrl: "/dashboard/settings/integrations/stripe"
   }
   ```

2. Update API endpoints for leads, deals, customers to wrap integration calls:
   ```typescript
   try {
     return await integrationService.createInPipedrive(...);
   } catch (error) {
     if (error instanceof CircuitOpenError) {
       const fallback = createUserFallbackResponse("pipedrive", "create_lead", "transient");
       return NextResponse.json(fallback, {status: 202}); // Accepted, will retry
     }
     // For critical endpoints, return user-friendly error
     const fallback = createUserFallbackResponse("pipedrive", "create_lead", categorizeError(error));
     return NextResponse.json(fallback, {status: 500});
   }
   ```

3. Key patterns:
   - Transient errors → status 202 (Accepted) with "waiting" message
   - Auth errors → status 401 with reconnect link
   - Validation errors → status 400 with specific field error
   - Permanent errors → status 500 with "contact support" message
   - Critical paths (lead creation) → queue for retry, return "queued" response

DO NOT expose internal errors to users (always use generic fallback messages).
DO NOT include stack traces or system details in user responses.
WHY: Security (no info leakage). UX (actionable guidance instead of cryptic errors). Reliability (users know what to do).
