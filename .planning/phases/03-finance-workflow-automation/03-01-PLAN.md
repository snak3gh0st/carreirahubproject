**Issues being addressed:**
- Manual invoice and contract generation (Finance team waste 10+ hours/week)
- No unified workflow tracking across QuickBooks and DocuSign
- Customer data inconsistency between financial systems
- No error recovery for failed integration steps

       - Log workflow start to IntegrationLog

       - Enhance processDealWon() method to orchestrate:
         a) Reconcile customer via Identity Mapper (ensure QuickBooks + DocuSign IDs)

         c) Generate contract in DocuSign via docusignService.createEnvelope()
         d) Create workflow tracking record in database (new WorkflowExecution model if needed, or use existing Deal fields)
         e) Log each step to IntegrationLog (service="WORKFLOW", action="INVOICE_CREATED"/"CONTRACT_SENT")

    3. **Error handling**:
       - Wrap each integration call in try/catch
       - Continue workflow even if one step fails (log error, mark step as failed)
       - Do NOT use Stripe (removed from Sprint 1 per PROJECT.md)
       - If QuickBooks fails, retry up to 3 times with exponential backoff
       - If DocuSign fails, mark contract as FAILED but continue (Finance can manually resend)

    4. **What to avoid and WHY**:
       - Do NOT create new database models unless absolutely necessary - use existing Deal.status or metadata fields for workflow state
       - Do NOT add Stripe integration - it was removed from Sprint 1 scope
       - Do NOT make workflow synchronous - use BullMQ queues for async processing to avoid Vercel 10s timeout
       - Do NOT fail entire workflow if one step fails - graceful degradation is critical
  </action>

  <action>
    Create workflow status tracking and manual retry capabilities:

    1. **Database schema (if not exists)**:
       - Add fields to Deal model for workflow tracking:
         - workflowStatus: PENDING | IN_PROGRESS | COMPLETED | FAILED
         - workflowStartedAt: DateTime
         - workflowCompletedAt: DateTime
         - workflowError: String (last error message)
         - invoiceCreatedAt: DateTime
         - contractSentAt: DateTime
       - Run `npx prisma db push` to apply schema changes (no migration needed for dev)

    2. **Create lib/services/workflow-status.service.ts**:
       - Class: WorkflowStatusService
       - Methods:
         - updateWorkflowStatus(dealId, status, error?) - Update Deal workflow fields
         - getWorkflowStatus(dealId) - Fetch current workflow state
         - retryFailedStep(dealId, step: "invoice" | "contract") - Retry specific failed step
       - Use existing services (quickbooksService, docusignService) for retries
       - Log all status changes to IntegrationLog

    3. **Create app/api/deals/[id]/workflow/route.ts**:
       - GET /api/deals/:id/workflow - Return current workflow status
       - POST /api/deals/:id/workflow/retry - Retry failed workflow step
       - Query IntegrationLog for workflow history (filter by dealId in payload)
       - Return JSON: { status, steps: [...], errors: [...], canRetry: boolean }

    4. **Integration with invoice-workflow.service.ts**:
       - Call workflowStatusService.updateWorkflowStatus() at each workflow step
       - Set workflowStatus=IN_PROGRESS when starting
       - Set workflowStatus=COMPLETED when all steps succeed
       - Set workflowStatus=FAILED + workflowError when critical step fails

    5. **What to avoid and WHY**:
       - Do NOT create a separate WorkflowExecution table if Deal model can hold the state - simpler schema is better
       - Do NOT make workflow status real-time - Finance dashboard can poll every 30s (no WebSocket needed)
       - Do NOT retry automatically indefinitely - limit to 3 retries per step, then require manual intervention
       - Do NOT expose customer PII in workflow status API - only show deal ID, status, and error messages
  </action>
