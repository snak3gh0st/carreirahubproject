    - Add `Subscription` model:
      ```prisma
      model Subscription {
        id              String   @id @default(uuid())
        customerId      String
        customer        Customer @relation(fields: [customerId], references: [id])

        stripeSubscriptionId  String   @unique
        stripePriceId        String   // Stripe price ID for the plan
        status              String   // active, canceled, past_due, trialing, etc.
        currentPeriodStart  DateTime
        currentPeriodEnd    DateTime
        cancelAtPeriodEnd   Boolean  @default(false)
        trialStart          DateTime?
        trialEnd            DateTime?

        amount              Decimal  @db.Decimal(10, 2)
        currency            String   @default("usd")
        interval            String   // month, year

        metadata            Json?

        createdAt           DateTime @default(now())
        updatedAt           DateTime @updatedAt

        @@index([customerId])
        @@index([stripeSubscriptionId])
        @@index([status])
      }
      ```

<task type="auto">
  <name>Task 2: Implement subscription lifecycle webhook handlers</name>
  <files>app/api/webhooks/stripe/route.ts, lib/services/stripe.service.ts</files>
  <action>
    Handle subscription lifecycle events via webhooks to keep database in sync.

    **Webhook event handlers (app/api/webhooks/stripe/route.ts):**

    **1. customer.subscription.created:**
    - Extract subscription from event.data.object
    - Check if subscription already exists in database (by stripeSubscriptionId)
    - If not exists, create Subscription record
    - If exists, update status
    - Log to IntegrationLog

    **2. customer.subscription.updated:**
    - Extract subscription from event
    - Update Subscription record in database with latest data:
      - status
      - currentPeriodStart
      - currentPeriodEnd
      - cancelAtPeriodEnd
      - amount (if price changed)
    - Log change to IntegrationLog

    **3. customer.subscription.deleted:**
    - Extract subscription from event
    - Update Subscription status to "canceled" in database
    - Set updatedAt timestamp
    - DO NOT delete record (keep for history)
    - Log to IntegrationLog

    **4. invoice.payment_succeeded (for subscriptions):**
    - Extract invoice from event
    - Check if invoice.subscription exists (not all invoices are subscriptions)
    - If subscription invoice:
      - Update subscription status to "active" (in case it was "past_due")
      - Update currentPeriodEnd if changed
    - Log to IntegrationLog

    **5. invoice.payment_failed (for subscriptions):**
    - Extract invoice and subscription
    - Update subscription status to "past_due"
    - Log to IntegrationLog
    - Note: Retry logic is in Plan 02-04 (Failed Payment Handling)

    **Service helper method (lib/services/stripe.service.ts):**
    - Add `syncSubscriptionFromStripe(stripeSubscriptionId: string)` method
    - Fetches subscription from Stripe API
    - Updates or creates Subscription record in database
    - Returns Subscription object
    - Use this in webhook handlers to ensure data consistency

    **What to avoid:**
    - Do NOT implement retry logic here (that's Plan 02-04)
    - Do NOT send email notifications (not in Sprint 1 scope)
    - Do NOT handle prorated changes yet (that's Task 3)
  </action>

    4. invoice.payment_succeeded updates subscription to active
    5. invoice.payment_failed marks subscription as past_due

    7. Test: Create subscription via Stripe dashboard, check database updated

<task type="auto">
  <name>Task 3: Implement subscription upgrades, downgrades, and cancellations</name>
  <files>app/api/stripe/subscription/[id]/route.ts, lib/services/stripe.service.ts</files>
  <action>
    Add API endpoints and service methods for subscription changes (upgrade, downgrade, cancel).

    **Subscription update endpoint (app/api/stripe/subscription/[id]/route.ts):**
    - Create PATCH endpoint at `/api/stripe/subscription/[id]`
    - Accept JSON: `{ action: 'change_plan' | 'cancel', priceId?: string, cancelAtPeriodEnd?: boolean }`
    - Validate subscription exists in database
    - Route to appropriate service method based on action

    **Service methods (lib/services/stripe.service.ts):**

    **1. changeSubscriptionPlan(subscriptionId: string, newPriceId: string):**
    - Fetch subscription from database to get stripeSubscriptionId
    - Update subscription in Stripe:
      ```typescript
      stripe.subscriptions.update(stripeSubscriptionId, {
        items: [{
          id: subscription.items.data[0].id,
          price: newPriceId
        }],
        proration_behavior: 'always_invoice'  // Prorated charges/credits
      })
      ```
    - Stripe automatically handles proration (creates invoice for difference)
    - Update Subscription record in database with new amount/price
    - Log to IntegrationLog
    - Return updated subscription

    **2. cancelSubscription(subscriptionId: string, immediate: boolean = false):**
    - Fetch subscription from database
    - If immediate=true:
      - Cancel immediately: `stripe.subscriptions.cancel(stripeSubscriptionId)`
      - Update database status to "canceled"
    - If immediate=false:
      - Cancel at period end: `stripe.subscriptions.update(stripeSubscriptionId, { cancel_at_period_end: true })`
      - Update database: cancelAtPeriodEnd = true
    - Log to IntegrationLog
    - Return updated subscription

    **Trial handling:**
    - Trials are handled automatically by Stripe (set trial_period_days at creation)
    - Webhook customer.subscription.updated fires when trial converts to active
    - No additional code needed (already handled in Task 2)

    **Error handling:**
    - Handle subscription not found (404)
    - Handle Stripe API errors (log and return 500)
    - Handle invalid price ID (400)

    **What to avoid:**
    - Do NOT implement custom proration logic (Stripe handles it)
    - Do NOT refund immediately on downgrade (Stripe credits account automatically)
    - Do NOT allow plan changes for canceled subscriptions (return 400)
  </action>
  <verify>
    1. PATCH /api/stripe/subscription/[id] with action=change_plan updates plan
    2. Proration calculated correctly by Stripe (check invoice in Stripe dashboard)
    3. PATCH with action=cancel and immediate=true cancels subscription immediately
    4. PATCH with action=cancel and immediate=false sets cancel_at_period_end
    5. Database updated after plan change
    6. Subscription webhooks fire after changes (handled by Task 2)
    7. Test: Upgrade plan → check prorated invoice created
    8. Test: Cancel subscription → check status updated
