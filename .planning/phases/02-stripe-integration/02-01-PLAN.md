<task type="auto">
  <name>Task 1: Create Stripe webhook endpoint with signature verification</name>
  <files>app/api/webhooks/stripe/route.ts</files>
  <action>
    Create POST endpoint at `/api/webhooks/stripe` following the QuickBooks webhook pattern.

    **Implementation:**
    - Import Stripe SDK and webhook validation utilities
    - Extract raw request body (required for signature verification)
    - Verify webhook signature using `STRIPE_WEBHOOK_SECRET` and Stripe SDK's `stripe.webhooks.constructEvent()`
    - Parse event type from `event.type`
    - Handle these event types initially (more added in later plans):
      - `payment_intent.succeeded` - Payment completed successfully
      - `payment_intent.failed` - Payment failed
      - `checkout.session.completed` - Checkout completed (for payment links)
    - Log all webhook events to `IntegrationLog` table with service="stripe", action=event.type
    - Return 200 OK immediately (Stripe expects response within 10s)
    - For events requiring processing, extract relevant data and process inline (don't queue yet - that's for Plan 4)

    **Security:**
    - Use Stripe SDK's built-in signature verification (NOT custom HMAC)
    - Stripe signature is in `stripe-signature` header
    - Return 401 if signature invalid
    - Return 400 if body parsing fails

    **Error handling:**
    - Catch all errors and log to IntegrationLog with status="ERROR"
    - Return 200 OK even on processing errors (to prevent Stripe retries for app logic failures)
    - Only return 4xx/5xx for signature failures or malformed requests

    **What to avoid:**
    - Do NOT use custom HMAC verification (use Stripe SDK's `constructEvent` method - it handles signature verification correctly)
    - Do NOT queue webhook processing yet (that's for Plan 4 with retry logic)
    - Do NOT process subscription events yet (handled in Plan 3)
    - Do NOT implement duplicate detection yet (that's Task 2)
  </action>

<task type="auto">
  <name>Task 2: Implement webhook deduplication and idempotency</name>
  <files>app/api/webhooks/stripe/route.ts, prisma/schema.prisma</files>
  <action>
    Add deduplication to prevent processing the same webhook event multiple times.

    **Database schema changes (prisma/schema.prisma):**
    - Add `StripeWebhookEvent` model:
      ```prisma
      model StripeWebhookEvent {
        id          String   @id @default(uuid())
        eventId     String   @unique  // Stripe event ID (e.g., "evt_...")
        eventType   String              // e.g., "payment_intent.succeeded"
        processed   Boolean  @default(false)
        processedAt DateTime?
        payload     Json                // Store full event payload
        error       String?             // Store error if processing failed
        createdAt   DateTime @default(now())

        @@index([eventId])
        @@index([eventType])
        @@index([processed])
      }
      ```
    - Run `npm run db:generate` after schema change
    - Run `npm run db:push` to apply schema (development) or create migration (production)

    **Webhook endpoint updates:**
    - After signature verification, extract Stripe event ID from `event.id`
    - Check if event already exists in database: `prisma.stripeWebhookEvent.findUnique({ where: { eventId: event.id } })`
    - If exists and `processed === true`: log "Duplicate event ignored" and return 200 OK immediately
    - If exists and `processed === false`: this is a retry of a failed event, proceed with processing
    - If doesn't exist: create record with `processed: false` before processing
    - After successful processing: update record with `processed: true, processedAt: now()`
    - If processing fails: update record with `error: errorMessage` but keep `processed: false` (allows retry)

    **Idempotency pattern:**
    - Use database transaction to prevent race conditions (multiple webhooks arriving simultaneously)
    - Pattern: `await prisma.$transaction(async (tx) => { check + create + process + update })`

    **What to avoid:**
    - Do NOT use Redis for deduplication (database is source of truth)
    - Do NOT delete old webhook records (keep for audit trail)
    - Do NOT fail webhook if database insert fails (log error but return 200 to prevent Stripe retries)
  </action>
