  <action>
    Add payment retry system using BullMQ queue with exponential backoff for failed payments.

    **Database schema changes (prisma/schema.prisma):**
    - Add `PaymentRetry` model:
      ```prisma
      model PaymentRetry {
        id                    String   @id @default(uuid())
        invoiceId             String?
        invoice               Invoice? @relation(fields: [invoiceId], references: [id])
        subscriptionId        String?
        subscription          Subscription? @relation(fields: [subscriptionId], references: [id])

        stripePaymentIntentId String?
        stripeInvoiceId       String?  // For subscription invoices

        failureReason         String
        failureCode           String?
        attempts              Int      @default(0)
        maxAttempts           Int      @default(3)
        nextRetryAt           DateTime?
        lastAttemptAt         DateTime?
        status                String   // pending, retrying, succeeded, failed, cancelled

        metadata              Json?

        createdAt             DateTime @default(now())
        updatedAt             DateTime @updatedAt

        @@index([invoiceId])
        @@index([subscriptionId])
        @@index([status])
        @@index([nextRetryAt])
      }
      ```
    - Add PaymentRetry relations to Invoice and Subscription models
    - Run `npm run db:generate` and `npm run db:push`

    **BullMQ Queue setup (lib/utils/queue.ts):**
    - Check if `paymentRetry` queue exists
    - If not, add new queue:
      ```typescript
      export const paymentRetryQueue = new Queue('paymentRetry', {
        connection: redis,
        defaultJobOptions: {
          attempts: 3,
          backoff: {
            type: 'exponential',
            delay: 60000  // Start with 1 minute
          },
          removeOnComplete: false,  // Keep for audit trail
          removeOnFail: false
        }
      });
      ```

    **Webhook updates (app/api/webhooks/stripe/route.ts):**
    - In `payment_intent.failed` handler:
      1. Extract Payment Intent and failure reason
      2. Extract invoice_id from metadata
      3. Create PaymentRetry record with status="pending"
      4. Enqueue retry job: `paymentRetryQueue.add('retryPayment', { paymentRetryId })`
      5. Log to IntegrationLog

    - In `invoice.payment_failed` handler (for subscriptions):
      1. Extract Stripe invoice and subscription
      2. Create PaymentRetry record linked to subscription
      3. Enqueue retry job
      4. Log to IntegrationLog

    **Service method (lib/services/stripe.service.ts):**
    - Add `retryPayment(paymentRetryId: string)` method:
      - Fetch PaymentRetry record from database
      - Increment attempts counter
      - If attempts >= maxAttempts: set status to "failed", return
      - Calculate next retry delay: 60s * (2 ^ attempts) (exponential backoff: 1min, 2min, 4min)
      - Attempt payment:
        - For one-time payments: Retrieve Payment Intent, check if still requires action
        - For subscriptions: Retrieve invoice, attempt to finalize/retry
      - If successful: update PaymentRetry status to "succeeded", update invoice/subscription
      - If failed: update lastAttemptAt, schedule next retry
      - Log to IntegrationLog

    **What to avoid:**
    - Do NOT retry immediately (use exponential backoff to avoid overwhelming customer)
    - Do NOT retry infinitely (max 3 attempts)
    - Do NOT email customer on every retry (that's Task 2)
    - Do NOT create new Payment Intent (retry same one)

    - Requires FINANCE role
    - Validates subscription exists and is past_due
    - Fetches latest failed invoice from Stripe
    - Attempts to retry invoice payment
    - Updates subscription status if successful
    - Returns retry result

    **Dunning management logic:**
    - Add `dunningLevel` field to Subscription model (optional):
      ```prisma
      // In Subscription model:
      dunningLevel Int @default(0)  // 0=none, 1=soft, 2=hard, 3=final
      ```
    - Update dunning level based on failed payment count:
      - 0 failed payments: dunningLevel = 0 (no action)
      - 1 failed payment: dunningLevel = 1 (soft reminder - not implemented in Sprint 1)
      - 2 failed payments: dunningLevel = 2 (hard warning - not implemented in Sprint 1)
      - 3+ failed payments: dunningLevel = 3 (final notice, consider cancellation)
    - Increment dunningLevel in `invoice.payment_failed` webhook
    - Reset dunningLevel to 0 in `invoice.payment_succeeded` webhook
