<objective>
Implement QuickBooks OAuth token refresh automation and create user-facing UI for manual token refresh to ensure continuous integration without user action.

Purpose: QuickBooks OAuth access tokens expire periodically (every 60 days). Currently, users receive errors when tokens expire and must manually reconnect via OAuth flow. This plan implements automatic refresh via cron job and provides UI for manual refresh when needed.

Output: Automated token refresh via cron endpoint, token expiration monitoring, user-facing token refresh UI in settings, graceful degradation when tokens are invalid.
</objective>

**QuickBooks OAuth Current State (from CLAUDE.md and codebase):**
- lib/services/quickbooks.service.ts already implements token refresh logic (methods starting at line 230)
- Access tokens stored in SystemConfig table (id="system")
- Token refresh happens on-demand when 401 error received (lib/services/quickbooks.service.ts:125-136)
- Problem: On-demand refresh only works if error occurs, doesn't prevent expiration scenarios

**Token Storage (SystemConfig model):**
- quickbooks_access_token: Current access token
- quickbooks_refresh_token: Refresh token (lasts longer, used to get new access token)
- quickbooks_token_expires_at: Timestamp of expiration

**Current Refresh Implementation:**
- QuickBooksService.refreshAccessToken() exists (line 230-293 in service)
- Makes OAuth request to https://oauth.platform.intuit.com/oauth2/v1/tokens/bearer
- Uses grant_type="refresh_token" with refresh_token and client credentials
- Updates database with new tokens and expiration time

**Files to Create/Modify:**

**Integration Points:**

## Decisions Made

1. **Daily Token Refresh via Cron**: Automatic refresh every day at 2 AM UTC
   - Rationale: Access tokens valid ~60 days, daily checks catch expiration early
   - Cost: One Vercel invocation per day (negligible)

2. **ADMIN-Only Endpoints**: Token status and manual refresh require ADMIN role
   - Rationale: Token information is sensitive, prevent non-admins from querying

3. **Logging to IntegrationLog**: All refresh attempts logged for monitoring
   - Rationale: Enables alerting, historical analysis, debugging failed refreshes

4. **Graceful Degradation**: If refresh fails, return structured error instead of throwing
   - Rationale: Prevents cron from crashing, UI can handle error responses
