## Technical Decisions

### Decision 1: Calculate Stats in Memory vs. Separate DB Queries
**Context**: Need to display financial summary and invoice list for customer

**Decision**: Fetch all invoices once, calculate stats in JavaScript
**Rationale**:
- Reduces database load (1 query instead of 5+)
- Faster for customers with < 100 invoices (typical case)
- Simpler code (no need to sync multiple query results)
- Invoice data needed for table anyway, so no extra data fetched
- Performance acceptable for target dataset size

### Decision 2: Server Component (Not Client Component)
**Context**: Need to display customer financial data

**Decision**: Keep as server component with server-side data fetching
**Rationale**:
- Follows established pattern in codebase (invoice page, customer list)
- No client-side interactivity needed (no sorting, filtering)
- Better performance (no client-side hydration)
- SEO-friendly (though not applicable for auth-protected dashboard)
- Simpler codebase (no useState, useEffect)

### Decision 3: Installment Summary as Separate Section
**Context**: Plan called for installment tracking

**Decision**: Created dedicated section above invoice table (not embedded in table)
**Rationale**:
- Better visual hierarchy (summary first, details below)
- Easier to scan key metrics without scrolling through table
- Matches existing pattern (invoice page has summary cards above table)
- Finance team can see payment status without analyzing table data

### Decision 4: No Pagination on Invoice Table
**Context**: Invoice table could have many rows for some customers

**Decision**: Show all invoices without pagination (as per plan guidance)
**Rationale**:
- Plan explicitly stated "Don't add pagination yet (premature)"
- Most customers have < 50 invoices (typical for this business)
- Horizontal scroll handles overflow on mobile
- Can add pagination later if needed (YAGNI principle)

## Lessons Learned

1. **Pattern Reuse**: Existing invoice page and customer list provided excellent patterns for styling, structure, and data fetching
2. **In-Memory Calculations**: For datasets < 100 records, in-memory calculations are simpler and faster than multiple DB queries
3. **Server Components**: Next.js 14 server components make data fetching very straightforward (no useEffect, no loading states)
4. **Prisma Include**: Using `include: { invoices: { orderBy: ... } }` fetches all related data in one query efficiently
